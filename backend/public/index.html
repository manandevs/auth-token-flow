What this project does

Implements a complete auth token flow with input validation, password hashing, JWT generation, and a protected profile route.

How it’s used

Signup → Login → call /api/profile with Authorization: Bearer <token>.

Interactive workspace for API routes

These are the backend endpoints and what they do. The frontend uses Axios and stores the JWT in localStorage for persistence.

Routes

  • POST /api/auth/signup — validate input, hash password (bcrypt), save user
  • POST /api/auth/login — check email, compare password, generate JWT (1d)
  • GET /api/profile — protected route (requires Bearer token)
Authorization: Bearer <your_jwt_token>

Example payloads

Signup

{ "name": "Abdul", "email": "abdul@example.com", "password": "12345678" }

Clear visualization of security basics

Passwords are never stored in plain text, JWT secrets come from .env, and token expiry is set to 1 day.

Security checklist

bcrypt hashing

Passwords are hashed during signup and never returned to the client.

JWT secret

Tokens are signed with JWT_SECRET from the backend .env.

Token expiry

JWT expires in 1d and protected routes verify the token before access.

Frontend flow

Frontend pages: /signup, /login, /dashboard (protected). Auth context loads token on refresh and fetches profile automatically.
